# IronRDP Web — Worker + OffscreenCanvas + softblit (the "best shape" render path)

## Goal

Run the entire IronRDP web session **off the main thread**, presenting via **softblit (WebGPU)** to
an **OffscreenCanvas**, with input/cursor/clipboard bridged over `postMessage`. This is the genuine
root fix for the two problems we hit on the main-thread path:

- **Back-pressure / "row-by-row" fill**: decode and present share the main thread and compete with
  the browser compositor; under load the client falls behind, acks lag, and the server downshifts to
  progressive banding. Moving decode+present to a worker frees the main thread; GPU present is cheap
  enough to coexist with decode on the worker thread.
- **Tearing**: Source A (client presenting mid-RDP-frame) is fixed by snapshotting at
  `FrameMarker::End`; Source B (server splitting one moment across brackets) is server-paced and only
  *minimized* by keeping the client from falling behind.

Why softblit + worker **compose**: GPU present is a cheap command submit, so it barely steals CPU
from decode — that's what lets decode own the worker thread. A CPU `put_image_data` present in the
worker would just recreate the contention.

## Current architecture (main-thread, as of `experiment/softblit-worker`)

- `<iron-remote-desktop>` custom element (`web-client/iron-remote-desktop/src/iron-remote-desktop.svelte`)
  owns the `<canvas>` and all DOM listeners.
- `RemoteDesktopService` (`.../services/remote-desktop.service.ts`) wraps the wasm: builds a
  `SessionBuilder`, calls `renderCanvas(HtmlCanvasElement)`, `connect()`, then `applyInputs(InputTransaction)`.
- The JS-facing `SessionBuilder`/`Session` surface is generated by a **trait + wasm-bindgen macro in
  the shared `iron-remote-desktop` crate** (also used by the VNC backend) — so canvas/input API
  changes are cross-cutting, not local to `ironrdp-web`.
- Rust `Session::run()` (`crates/ironrdp-web/src/session.rs`) is an async `select!` loop over
  WebSocket I/O + input channel + timers; presents via `Canvas` (`canvas.rs`: softblit GPU path with
  Canvas2D fallback). `requestAnimationFrame` is available in workers, so the present loop ports.
- Build: `wasm-pack build --target web` (ES module — usable as a module worker), bundled by Vite
  (`vite-plugin-wasm`, `vite-plugin-top-level-await`).
- **No worker / OffscreenCanvas usage anywhere today.**

## Key decisions

1. **Worker path is opt-in + GPU-only.** Add a `?renderer=worker` switch (mirrors the existing
   `?protocol=vnc`). The main-thread softblit/Canvas2D path stays as the default and the fallback, so
   the demo is never broken and we can A/B. In the worker, require WebGPU; if unavailable, the worker
   reports it and the app falls back to the main-thread path (no `OffscreenCanvasRenderingContext2d`
   reimplementation needed).
2. **Whole session in the worker** (WebSocket connect incl. CredSSP/gateway, decode, present). Main
   thread keeps only: DOM canvas + listeners, and a thin proxy.
3. **Cursor / clipboard / warning callbacks** already go through JS `Function`s — in worker mode the
   worker supplies `postMessage` wrappers; the main thread applies the DOM cursor / clipboard. No DOM
   access from the worker.
4. **Input** is serialized as plain event data (`{type,x,y,button,scancode,...}`) main→worker; the
   `InputTransaction`/`DeviceEvent` objects are built **in the worker**.

## Stages (each ends build-green; demo runnable via the default path until the worker path is wired)

- **S1 — Rust: `RenderTarget` plumbing.** `RenderTarget { Html(HtmlCanvasElement),
  Offscreen(OffscreenCanvas) }` threaded through `SessionBuilderInner` → `Session` → `Canvas::new` →
  resize. `canvas.rs` GPU path maps to `softblit::SurfaceTarget::{Canvas,OffscreenCanvas}`; offscreen
  is GPU-only (errors if no WebGPU). web-sys `OffscreenCanvas` feature. *Verify: `cargo check
  --target wasm32`.*
- **S2 — shared trait + macro: `renderOffscreenCanvas`.** Add to the `iron-remote-desktop`
  SessionBuilder trait **with a default impl** (so the VNC backend doesn't break) + the wasm-bindgen
  macro export. *Verify: wasm32 check; VNC unaffected.*
- **S3 — TS worker module.** `iron-remote-desktop-rdp/src/worker.ts`: `init()` wasm, on
  `{offscreen,config}` build the session against the transferred OffscreenCanvas, run it; bridge
  cursor/clipboard/warning out and input in via `postMessage`. Vite `?worker` import.
- **S4 — `RemoteDesktopService` worker proxy.** When `renderer=worker`: `transferControlToOffscreen`,
  spawn worker, proxy `connect`/`applyInputs`/`resize`/clipboard/cursor as messages; serialize raw
  input events. Keep the direct path for the default.
- **S5 — custom element + build wiring.** `?renderer=worker` switch; ensure the worker entry bundles
  the wasm correctly (Vite worker + `vite-plugin-wasm`).
- **S6 — boundary snapshot, GPU-side.** Upload dirty rects to the softblit texture at
  `FrameMarker::End` (Source-A fix), present on the worker's rAF. Confirm tearing floor = Source B.
